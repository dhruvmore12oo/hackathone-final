1.2  Elevator Pitch
FlowERP is a modern, role-based Mini ERP platform that digitally orchestrates the complete manufacturing lifecycle — from the moment a customer places an order to the instant finished goods are delivered. By unifying Products, Sales, Procurement, Manufacturing, and Inventory into a single, real-time system, FlowERP eliminates the spreadsheet chaos, WhatsApp coordination, and stock blindness that cripple growing businesses like Shiv Furniture Works.

1.3  Problem Statement
Growing manufacturing companies operate across 5+ disconnected tools: Excel inventory sheets, WhatsApp-based purchase requests, paper BoMs, manual stock registers, and verbal work-order assignments. This creates four critical failure modes:
Sales teams oversell — committing stock that doesn't exist or isn't available
Procurement fires late — purchase orders placed only after production halts
Manufacturing operates blind — operators don't know what to build or in what sequence
Management has zero visibility — no live KPIs, no pending order alerts, no shortage warnings

1.4  Why This Matters
A single missed delivery costs a furniture company 3–8% of annual customer lifetime value. Stock mismatches cause 23% production downtime on average in SME manufacturing. Procurement delays add 2–4 weeks to lead times. FlowERP solves all three simultaneously.

  05    DETAILED MODULE SPECIFICATIONS

5.1  Authentication & RBAC
Purpose
Secure the system via JWT-based login with role-based access. Each user has exactly one role that determines which modules are visible and editable.
User Stories
As a user, I can log in with email and password so I can access my permitted modules
As an Admin, I can create users and assign roles
As a Sales User, I can only see the Sales module and Dashboard
Business Rules
Roles: Admin, Sales User, Purchase User, Manufacturing User, Inventory Manager, Business Owner
JWT access token expires in 8 hours; refresh token in 7 days
Admin has unrestricted access to all modules including Audit Logs
Non-admin users cannot see Audit Logs or User Management
Failed login attempts ≥5 lock account for 15 minutes
Validation Rules
Email must be valid format and unique
Password minimum 8 characters
Role must be one of the defined enum values
Token must be validated on every protected API request
Edge Cases
Admin deletes their own account → system must block if they are the last Admin
User role changed mid-session → frontend must re-fetch permissions on next navigation
UI Requirements
Clean login card: logo, email, password, Login CTA — no clutter
Role badge displayed in user avatar dropdown
Forbidden module navigation shows a 403 empty state, not a broken page

5.2  Executive Dashboard
Purpose
The first screen judges see. Delivers real-time operational KPIs across all modules in a premium, information-dense layout that communicates business health instantly.
User Stories
As a Business Owner, I see live KPIs for Sales, Purchase, Manufacturing, and Inventory on one screen
As an Inventory Manager, I see low-stock alerts immediately on the dashboard
Business Rules
KPI cards refresh every 30 seconds or on manual refresh
Low-stock threshold is configurable per product (default: 10 units)
Manufacturing orders delayed beyond scheduled date are flagged as red
Validation Rules
All counts must be real-time from DB, not cached older than 30 seconds
Edge Cases
No data state: dashboard shows skeleton cards with setup prompts
All modules empty: show a demo seeder prompt for judges
UI Requirements
6 KPI cards in 3×2 grid: Total SOs, Pending Deliveries, Total POs, Pending Receipts, Active MOs, Low Stock Alerts
Inventory health bar chart: On-Hand vs Reserved per top 5 products
Manufacturing progress ring chart: Draft / In-Progress / Completed MOs
Procurement alerts table: top 5 most critical low-stock products
Recent activity feed: last 10 audit log entries
Order analytics: 7-day SO and PO creation trend line chart

5.3  Products Module
Purpose
Central catalog of all products. Each product carries pricing, stock quantities, and procurement configuration that drives the entire ERP flow.
User Stories
As an Admin/Owner, I can create products with sales price, cost, and procurement config
As an Inventory Manager, I can view on-hand, reserved, and free-to-use quantities per product
As a Sales Manager, I can see free-to-use quantity before confirming an order
Business Rules
Free-To-Use Qty = On-Hand Qty − Reserved Qty (computed, never stored)
Procurement Type: Purchase (uses vendor) or Manufacturing (uses BoM)
A product can only be deleted if it has no open Sales Orders, Purchase Orders, or Manufacturing Orders
Product linked to a BoM is classified as a 'Manufactured' product
Validation Rules
Sales Price and Cost Price must be positive decimals
On-Hand Qty must be non-negative integer
Procurement Type required if 'Procure on Demand' is enabled
If Procurement Type = Purchase, Vendor field becomes required
If Procurement Type = Manufacturing, BoM must exist for the product
Edge Cases
Product has zero stock and open SOs — deletion blocked with message
BoM deleted while product references it — product's procurement type must reset
UI Requirements
Product list: searchable, filterable by category and procurement type
Product detail panel (side drawer) shows live stock quantities with color-coded health bar
Inline edit for price fields
Stock quantity displayed as three chips: On-Hand (blue), Reserved (amber), Free (green)

5.4  Inventory Module
Purpose
Tracks every stock movement with a full ledger. Provides manual adjustment capability with reason codes. Acts as the system of record for all quantity changes.
User Stories
As an Inventory Manager, I can view the complete stock ledger with source references
As an Admin, I can manually adjust stock quantities with a reason code
Business Rules
Every stock change (from any module) writes a Stock Ledger entry
Ledger entry includes: product, date, qty_before, qty_change, qty_after, source_module, source_ref, user
Manual adjustments require a reason text (min 10 characters)
Validation Rules
Manual adjustment qty cannot bring On-Hand below zero
Reason code is mandatory for manual adjustments
Edge Cases
Adjustment that would make On-Hand negative → blocked with warning
UI Requirements
Stock ledger as a filterable, paginated table (filter by product, date range, source module)
Manual adjustment modal with product selector, qty (+/−), and reason field
Per-product stock history drawer with timeline visualization

5.5  Sales Orders
Purpose
Manages customer demand. Drives stock reservation and procurement triggers. Primary entry point for the MTS/MTO decision logic.
User Stories
As a Sales Manager, I can create a Sales Order with customer and product lines
As a Sales Manager, I can confirm an order which reserves stock and triggers procurement if needed
As a Sales Manager, I can mark an order as delivered (partial or full)
Business Rules
States: Draft → Confirmed → Partially Delivered → Fully Delivered | Cancelled
On Confirm: system checks Free-to-Use Qty per line item
If Free-to-Use Qty ≥ ordered qty: reserve stock (MTS path)
If Free-to-Use Qty < ordered qty and product is MTO: trigger procurement for shortage qty
Cancelled orders release reserved stock
Delivered orders permanently reduce On-Hand
Validation Rules
At least one order line required
Ordered qty must be positive integer
Customer name is required
Cannot confirm a Cancelled order
Edge Cases
Product deleted after SO confirmed → SO line shows 'Product Archived' warning
Partial delivery on multi-line order — each line has its own delivery status
UI Requirements
SO list with status badges (color-coded), search, date filter
SO detail page: header info + line items table + procurement action panel
Inline delivery input: 'Deliver X of Y' per line
Procurement trigger result shown as dismissible toast: 'PO/MO auto-created for 15 units'

5.6  Purchase Orders
Purpose
Manages vendor-based stock replenishment. Can be manually created or auto-generated by the procurement engine.
User Stories
As a Purchase Manager, I can create and confirm Purchase Orders
As a Purchase Manager, I can receive goods (partial or full) and see stock update automatically
Business Rules
States: Draft → Confirmed → Partially Received → Fully Received | Cancelled
On Full/Partial Receipt: On-Hand Qty increases by received quantity
Auto-created POs are linked to the originating Sales Order via source_so_id
Cancelled POs do not affect stock
Validation Rules
At least one PO line required
Received qty cannot exceed ordered qty
Vendor field required on confirmation
Edge Cases
Goods received exceeding ordered qty → system warns, prevents receipt above ordered qty
UI Requirements
PO list showing auto-created badge for system-generated orders
PO detail with receive goods action (inline qty input per line)
Vendor dropdown with quick-add capability

5.7  Bill of Materials (BoM)
Purpose
Defines the recipe for manufactured products — what components are needed, in what quantities, and what operations transform them into finished goods.
User Stories
As a Manufacturing Manager, I can create a BoM for a product with components and quantities
As a Manufacturing Operator, I can view the BoM to understand what I need before starting production
Business Rules
A product can have only one active BoM
BoM components are validated against the Products catalog
Operations define the steps in sequence (Assembly, Painting, Packing, etc.)
Each operation has a Work Center and estimated duration
Validation Rules
Finished product field required
At least one component required
Component qty must be positive
A product cannot be both a finished good and a component of its own BoM (circular reference)
Edge Cases
Circular BoM reference (product A uses product A as component) → system must detect and block
UI Requirements
BoM builder with drag-to-reorder component lines
Operations section with work center dropdown and duration input
Preview of BoM cost (sum of component costs × quantities)

5.8  Manufacturing Orders
Purpose
Converts raw material components into finished goods using BoM-driven work orders. Core production tracking module.
User Stories
As a Manufacturing Manager, I can create a Manufacturing Order for a product and quantity
As an Operator, I can view and execute my work orders step by step
As a Manager, I can mark an MO complete and see stock auto-update
Business Rules
States: Draft → In Progress → Completed | Cancelled
On Create: system fetches BoM and generates component list and work orders
On Start: component stock is reserved
On Complete: components deducted from On-Hand, finished goods added to On-Hand
Auto-created MOs are linked to originating Sales Order via source_so_id
Validation Rules
Product must have an active BoM to create an MO
Quantity must be positive
Cannot complete MO if component stock is insufficient
Edge Cases
Insufficient components on MO start → warning with shortage detail, operator can still proceed with override
MO cancelled after components reserved → reservation released
UI Requirements
Kanban-style MO board: Draft / In Progress / Completed columns
MO detail: progress bar showing work order completion (e.g., 2/3 steps done)
Work order execution timeline with start/complete buttons
Component checklist with available vs required quantities

5.9  Procurement Automation Engine
Purpose
The intelligence layer that automatically creates Purchase Orders or Manufacturing Orders when a Sales Order confirms and a shortage is detected, implementing MTO strategy without manual intervention.
User Stories
As a Sales Manager, when I confirm an MTO order with insufficient stock, the system auto-creates a PO or MO
As a Purchase Manager, I can see auto-created POs flagged distinctly from manual ones
Business Rules
Procurement trigger fires synchronously on Sales Order confirmation
Shortage qty = Ordered Qty − Free-to-Use Qty (only positive shortages trigger procurement)
Procurement Type on product determines whether PO or MO is auto-created
Auto-created orders appear with source_type = 'auto' and source_so_id reference
One trigger per order line (not one per order)
Validation Rules
Product must have Procure on Demand = true for auto-trigger to fire
Procurement Type must be set before auto-trigger can execute
Edge Cases
Two order lines for same product on one SO → procurement calculated per line, consolidated into one PO/MO
UI Requirements
Procurement trigger result shown as real-time banner on SO confirmation
Auto-created orders show 'Auto-Generated' badge with link to source SO

5.10  Audit Logs
Purpose
Immutable, comprehensive log of every significant system event. Provides full traceability for compliance, debugging, and business review.
User Stories
As an Admin, I can see a searchable, filterable log of all system events
As a Business Owner, I can see who changed what and when
Business Rules
Audit log entries are append-only (never updated or deleted)
Log entry schema: id, timestamp, user_id, user_name, module, action, entity_id, entity_type, field_changed, old_value, new_value, ip_address
Events logged: all CRUD operations, status transitions, stock adjustments, login/logout, procurement triggers
Validation Rules
Log entries cannot be deleted or modified via any API
Edge Cases
System crash during log write → transaction must be atomic (log + original action both succeed or both fail)
UI Requirements
Full-width filterable table: filter by module, action, user, date range
Search by entity ID or description
Color-coded action badges: CREATE (green), UPDATE (blue), DELETE (red), STATUS_CHANGE (amber)
Export to CSV for Admin users

6.2  Key Constraints & Indexes
products.sku — UNIQUE NOT NULL
products.on_hand_qty — CHECK (on_hand_qty >= 0)
products.reserved_qty — CHECK (reserved_qty >= 0)
boms — UNIQUE (product_id) where is_active = true (one active BoM per product)
bom_lines — no self-reference: component_product_id ≠ bom.product_id
stock_ledger — append-only enforced at application layer (no UPDATE/DELETE exposed in API)
audit_logs — append-only enforced at application layer
INDEX on: sales_orders(status), purchase_orders(status), manufacturing_orders(status), stock_ledger(product_id, date), audit_logs(module, timestamp)

  09    AI FEATURES

9.1  Smart Reorder Quantity Advisor
On any product detail page, an AI button analyzes the product's 90-day SO history and current stock trends to recommend an optimal reorder quantity and frequency. Implementation: send SO history array to Claude API with a prompt asking for reorder recommendation in JSON format. Response displayed as a suggestion card with reasoning.
Wow factor: judges see AI actively analyzing their own demo data
Implementation: ~40 lines of code — fetch SO history → call Claude API → parse JSON response → display card
Demo moment: create multiple SOs for Wooden Tables, then click 'Get AI Reorder Suggestion' and watch it recommend restocking 25 units based on weekly demand of ~5

9.2  Natural Language Audit Log Search
In the Audit Logs module, instead of using complex filters, the user types a plain-English question: 'Who changed the Wooden Table price last week?' or 'Show me all auto-procurement triggers for Dining Tables.' The system sends the query + recent log data to Claude API which returns the matching entries.
Wow factor: turns a boring log table into an intelligent business intelligence tool
Implementation: capture input → send to Claude with log schema context → parse returned filter criteria → apply to log query
Demo moment: type 'Show me everything that happened when we confirmed Sales Order #1' and watch the system surface the entire chain of events

9.3  AI Manufacturing Load Planner
When a Manufacturing Order is created, an AI panel suggests the optimal scheduling — which work center to assign, estimated completion time based on current MO queue, and whether procurement needs to happen before the MO can start. Implementation: send current MO queue and component availability to Claude API with a scheduling prompt.
Wow factor: shows AI as a proactive business advisor, not just a chatbot
Implementation: ~50 lines — fetch active MO queue + component stock → call Claude API → display scheduling recommendation
Demo moment: create an MO for 15 Dining Tables and let AI say 'Assembly line is at 80% capacity — recommend scheduling for Wednesday, but note that you are 5 Wooden Tops short — trigger procurement now'
